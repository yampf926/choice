# choice 작업 지침

## 프로젝트 개요
- 이 프로젝트는 Java Swing 기반의 단일 실행 파일 구조(`src/Main.java`)로 만든 선택형 스토리 게임이다.
- 현재 핵심 로직은 장면(Scene) 정의, 상태(GameState), 선택지(Choice), 이미지 로딩, 대사 애니메이션까지 모두 `Main.java`에 들어 있다.
- 화면 비율은 모바일 세로형(`9:16`)을 기준으로 강제되고, 배경/캐릭터/오버레이 레이어를 겹쳐서 연출한다.

## 작업 우선순위
- 먼저 `src/Main.java`를 읽고 현재 장면 흐름과 상태값 변화를 이해한 뒤 수정한다.
- 스토리 추가나 수정은 기존 `scenes.put(...)` 패턴, `Choice` 생성 패턴, `GameState` 갱신 패턴을 그대로 따른다.
- UI 수정은 `layoutSceneLayers()`, `refreshImages()`, `setImage()`의 연동을 깨지 않도록 최소 범위로 한다.
- 새 이미지나 씬을 추가할 때는 코드 수정 전에 실제 에셋 파일명이 `assets/images`와 일치하는지 확인한다.

## 코드 구조 규칙
- 장면 추가 시 다음 필드를 일관되게 채운다: 제목, 챕터, 내레이션, 화자, 대사, 배경 이미지, 캐릭터 이미지, 선택지, `onEnter`.
- 선택지가 상태를 바꿔야 하면 `Choice(..., effect)` 또는 `Choice(..., effect, visibility)` 생성자를 사용한다.
- 분기 잠금/해금이 필요하면 `GameState`에 불린 또는 카운터를 추가하고, 선택지 `visibility`로 제어한다.
- 화면 연출 변경은 기존 톤을 유지한다: 어두운 밤 배경, 반투명 패널, 강조색 `ACCENT` 중심.
- 리팩터링은 가능하지만, 현재 저장소는 단일 파일 중심이므로 큰 구조 분해는 요청이 있을 때만 진행한다.

## 에셋 규칙
- 이미지 경로는 상대 경로 `assets/images/<fileName>`를 사용한다.
- 코드에 참조하는 파일명은 `assets/images/IMAGE_LIST.txt` 및 실제 파일 목록과 맞아야 한다.
- 없는 파일명을 참조하면 UI는 폴백 텍스트만 보이므로, 장면 추가 후 반드시 파일명 오타를 점검한다.
- 캐릭터 이미지는 비율 유지 `contain`, 배경 이미지는 꽉 채우는 `cover` 방식으로 스케일된다. 이를 전제로 원본 이미지를 고른다.

## 텍스트와 인코딩
- 이 프로젝트는 한국어 텍스트 비중이 높다. 문자열 수정 시 한글이 깨지지 않도록 파일 인코딩을 특히 조심한다.
- 기존 텍스트가 깨져 보이면 무작정 전체 치환하지 말고, 현재 편집기/빌드 환경의 인코딩 설정부터 확인한다.
- 스토리 문체는 미스터리, 오컬트, 조사극 톤을 유지하고, 각 사건은 "오해를 진실처럼 보이게 만드는 구조"를 유지한다.

## 검증 기준
- 수정 후에는 최소한 컴파일이 되는지 확인한다.
- 이 프로젝트의 컴파일 산출 위치는 `out/production/choice/`로 본다.
- 새 장면을 넣었다면 다음을 확인한다: 진입 가능 여부, 버튼 노출 조건, 다음 장면 연결, 상태값 반영, 이미지 표시.
- `GameState.unlockClue()`처럼 현재 의미가 약한 메서드는 향후 확장 가능성을 고려하되, 기존 호출부 동작은 유지한다.
- `out/`은 빌드 산출물이다. 소스 수정은 `src/` 기준으로 하고, 산출물은 직접 편집하지 않는다.
- 모듈 설정으로 지정된 컴파일 경로인 `out/production/choice/` 외의 위치에는 `.class` 파일이나 별도 클래스 산출 문서를 만들지 않는다.

## 피해야 할 작업
- 사용자가 요청하지 않았는데 UI 구조를 전면 재설계하지 않는다.
- 사용자가 요청하지 않았는데 `Main.java`를 무리하게 다중 파일 구조로 분해하지 않는다.
- 이미지 파일명을 추측해서 코드만 바꾸지 않는다. 항상 실제 파일 존재 여부를 먼저 확인한다.
- 기존 사용자가 만든 변경사항을 되돌리지 않는다.
